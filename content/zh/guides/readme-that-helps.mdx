## 开始之前

你将为一个真实项目整理 README。完成后，一位第一次接触仓库的同学应能在十分钟内回答：项目是什么、当前做到哪里、怎样运行、怎样验证、去哪里继续了解。

README 不是宣传页，也不是所有文档的合集。它是项目的入口地图。

## 1. 用一句话说明项目

第一段应包含目标用户、解决的问题和产品形态。例如：

> 面向燕中校友与在校生的科创知识库，以学习路径、实作指南和真实项目连接学习与共建。

避免“功能强大”“智能高效”等无法验证的形容词。项目仍在规划时，明确写“计划建设”，不要把目标描述成现状。

## 2. 说明当前状态

用三到六条列出已经能够运行的能力，再单独写“暂不包含”或“下一阶段”。读者需要区分事实、计划和设想。

状态信息比长篇愿景更能建立信任。旧路线如果已被放弃，应删除或标记为历史决定。

## 3. 给出可复现的启动步骤

启动说明至少包含：

1. 运行环境与版本范围
2. 安装依赖
3. 配置示例环境变量
4. 启动命令
5. 本地访问地址或成功信号

```bash
git clone REPOSITORY_URL
cd PROJECT_NAME
pnpm install
pnpm dev
```

不要把真实密钥写进示例。需要配置时提供 `.env.example`，并解释每个变量的用途。

## 4. 写清验证方式

告诉贡献者怎样证明改动没有破坏项目：

```bash
pnpm typecheck
pnpm lint
pnpm build
```

只列仓库真实存在、当前可以通过的命令。需要数据库、账号或外部服务的检查要说明前置条件。

## 5. 建立文档入口

README 保留最短路径，把架构、安全、部署和内容规范链接到 `docs/`。链接文字应描述目标，例如“部署指南”，不要连续使用模糊的“点击这里”。

每次重命名文档后检查链接；失效链接会直接破坏新贡献者的第一条路径。

## 6. 从陌生人的角度复核

请一位没有参与开发的人按照 README 操作。记录对方停住的位置，而不是立即口头补充。需要口头解释的部分，通常正是文档缺口。

## 完成检查

- [ ] 开头能准确说明用户、问题和项目形态
- [ ] 已实现能力与未来计划分开描述
- [ ] 启动命令在干净环境中验证过
- [ ] 质量检查命令真实存在并能运行
- [ ] 没有密钥、个人信息或内部地址
- [ ] 文档链接有效，读者知道下一步去哪里

## 下一步

把 README 改进放在独立分支中，通过 Pull Request 邀请维护者复核。至此，你已经完成从认识项目、提交改动到改善协作入口的一条完整开源路径。
