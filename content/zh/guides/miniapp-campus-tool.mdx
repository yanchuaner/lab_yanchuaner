## 开始之前

你将做出一个可在微信开发者工具中运行的校园活动清单：用户能查看活动、按日期筛选，并把活动标记为关注。首版只使用本地假数据，不采集姓名、班级或联系方式。

准备微信开发者工具。没有正式 AppID 时可使用工具提供的测试能力；界面名称可能随版本变化，以官方文档为准。

## 1. 先定义最小范围

首版只保留三个动作：浏览活动、切换日期、关注或取消关注。登录、聊天、推送和后台发布都不是这次练习的目标。

一条活动数据包含：

```js
{
  id: "club-open-day",
  title: "社团开放日",
  date: "2026-07-20",
  place: "综合楼一层",
  followed: false
}
```

使用稳定的字符串 `id`，不要用数组位置标识活动。

## 2. 建立页面状态

在页面的 `data` 中保存活动数组和当前日期。派生出来的“筛选结果”应通过函数计算，不再保存一份容易失去同步的副本。

```js
Page({
  data: { selectedDate: "2026-07-20", events: [] },
  onLoad() { this.setData({ events: demoEvents }) }
})
```

成功信号：页面启动后能稳定显示假数据，刷新后没有脚本错误。

## 3. 渲染可扫描的活动列表

每个活动条目优先展示标题、日期和地点。关注按钮必须有文字状态，不能只靠颜色或图标区分。

列表使用活动 `id` 作为 key。空日期应显示“当天暂无活动”，而不是留下空白页面。

## 4. 实现日期筛选

日期变化时只更新 `selectedDate`，随后从原始数组计算匹配项。先验证没有活动、只有一项和多项三种情况。

避免每次渲染都修改原数组；筛选应该是无副作用的读取操作。

## 5. 更新关注状态

点击按钮时，从事件参数取得活动 `id`，使用 `map` 返回新数组：

```js
const events = this.data.events.map(event =>
  event.id === id ? { ...event, followed: !event.followed } : event
)
this.setData({ events })
```

成功信号：只改变被点击的活动，切换日期后状态仍与数组一致。

## 6. 处理本地持久化

原型可以把关注 ID 保存到本地存储，但不要把本地存储当成跨设备账号系统。加载时若数据损坏，应回退为空数组，而不是让页面崩溃。

## 7. 检查手机体验

- 主要按钮触控区域足够大
- 长活动标题能够换行
- 列表滚动时日期筛选仍容易找到
- 深色系统设置下文字依然可读
- 加载、空状态和失败状态都有明确说明

## 8. 数据与隐私边界

本练习只使用公开活动信息和假数据。不要把真实学生信息、内部群聊内容或未公开日程写进仓库。未来接 API 时，也应只请求当前页面需要的字段。

## 9. 完成检查

- [ ] 能按日期看到正确活动
- [ ] 关注操作只改变目标条目
- [ ] 无活动时有清晰空状态
- [ ] 关闭并重开后本地关注可恢复
- [ ] 仓库中没有 AppSecret、个人信息或真实名单

## 下一步

把数据读取封装为独立模块，并为筛选函数写最小单元测试。只有真实用户确认这个流程有价值后，再设计登录、云端同步或消息提醒。
